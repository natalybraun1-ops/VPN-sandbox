# Песочница VPN: дизайн продукта и архитектуры

Дата: 2026-06-12
Техническое имя проекта: `vpn-sandbox`
Пользовательское имя: "Песочница VPN"

## 1. Продуктовая модель

"Песочница VPN" - Windows-приложение для контроля сетевого поведения выбранных пользователем программ. Пользователь запускает браузеры и приложения обычным способом: с рабочего стола, панели задач, меню "Пуск" или ярлыков. Программа не является обязательным лаунчером. Она работает как фоновый контроллер правил.

В продукте есть один набор настроек и до двух зон:

- **VPN-зона**: выбранные `.exe` и их связанные дочерние процессы могут работать только при включенном VPN и совпадении страны с активным VPN-профилем. Город/регион отображается только как информационный маркер и не влияет на блокировку.
- **Прямая зона**: выбранные `.exe` и их связанные дочерние процессы должны идти напрямую в обход VPN. Подтверждение обхода может идти от системного компонента, от настроек VPN-клиента или от диагностического probe.

Первый запуск поддерживает три сценария:

- только VPN-контроль;
- только прямой обход VPN;
- две зоны.

Отключенная зона не применяет правила и не показывает предупреждения, но сохраняет свои настройки. Неконтролируемые приложения игнорируются и не показываются в интерфейсе.

При нарушении правил управляемое приложение блокируется на старте или получает предупреждение во время работы. Реакция на нарушение настраивается отдельно для каждой зоны:

- закрыть сразу;
- закрыть через 10 секунд;
- закрыть через 20 секунд;
- закрыть через 30 секунд;
- только предупреждать, если пользователь явно подтвердил, что понимает риск.

## 2. Архитектура

Техническая основа первой версии: **PyQt6-приложение + Windows service + native WFP/сетевой компонент**.

### PyQt6-приложение

Отвечает за пользовательскую часть:

- мастер первого запуска;
- настройки зон;
- списки управляемых приложений;
- VPN-профили;
- прямые профили;
- журнал событий;
- диагностические отчеты;
- проверку обновлений;
- трей;
- мини-индикатор поверх окон.

### Windows service

Работает в фоне и стартует вместе с Windows. Служба:

- применяет правила;
- следит за процессами;
- проверяет уже запущенные процессы при старте;
- получает сетевые события Windows;
- общается с WFP/native-компонентом;
- проверяет состояние зон;
- сообщает UI о статусах и событиях.

Служба является источником истины по статусу контроля. UI не применяет сетевые правила напрямую.

### Native WFP/сетевой компонент

Компонент дает фундамент для настоящего контроля:

- привязка сетевых потоков к процессам;
- раздельная маршрутизация и/или блокировка управляемых приложений;
- контроль связанных дочерних процессов;
- защита от утечек в рамках управляемых зон.

При сбое компонент не должен ломать общий интернет пользователя. Вместо этого зоны переходят в статус "контроль недоступен / требуется внимание".

### IPC

UI и служба общаются через локальный защищенный IPC-канал. UI передает настройки и получает статусы. Служба валидирует настройки перед применением.

### macOS позже

UI, база настроек и модель зон должны быть переносимыми. Для macOS нужен отдельный backend на базе Network Extension, Packet Filter или другого системного механизма. Linux не планируется.

## 3. Калибровка и профили

Первый запуск идет через мастер выбора сценария:

- "Только VPN-контроль";
- "Только прямой обход VPN";
- "Две зоны".

### VPN-профили

Для VPN-зоны мастер просит пользователя включить VPN в нужном состоянии и затем сканирует:

- внешний IP;
- страну;
- город/регион как информационный маркер;
- вероятный VPN-клиент;
- вероятный протокол;
- сетевой адаптер;
- DNS;
- маршруты;
- уровень уверенности распознавания.

Страна становится строгим условием профиля. Город не влияет на блокировку. Профиль получает автоматическое имя, например:

- "Германия · WireGuard";
- "Нидерланды · OpenVPN";
- "Турция · неизвестный протокол";
- "VPN-профиль 2", если данных мало.

Пользователь может переименовать профиль. Можно хранить несколько VPN-профилей, но активным для VPN-зоны назначается только один. Остальные профили запасные и не активируются автоматически. Ненужные профили можно удалить.

### Прямые профили

Если сценарий включает прямую зону, мастер обязательно делает снимок состояния без VPN. Пользователь выключает VPN, приложение сканирует baseline прямого подключения и при необходимости сравнивает его с VPN-состоянием.

Можно хранить несколько прямых профилей для разных сетей: дом, офис, мобильный интернет. Активный профиль выбирается вручную. Автоматического обновления baseline нет, чтобы случайно не сохранить VPN-состояние как прямое.

Имена прямых профилей назначаются автоматически:

- "Прямое подключение";
- "Прямое подключение 2";
- "Прямое подключение 3".

Пользователь может переименовать и удалить профиль.

### Распознавание VPN

Программа должна широко распознавать VPN-протоколы и клиенты:

- WireGuard;
- OpenVPN;
- IKEv2;
- L2TP;
- PPTP;
- SSTP;
- Outline;
- Amnezia;
- Shadowsocks;
- Clash;
- V2Ray/Xray;
- похожие туннельные клиенты.

Редкие случаи могут определяться предположительно. Если точности недостаточно, UI показывает "туннель обнаружен, протокол неизвестен".

## 4. Правила приложений и процессов

Пользователь добавляет приложения в зоны двумя способами:

- из списка найденных браузеров и популярных приложений;
- вручную через выбор `.exe`.

В MVP правило применяется ко всему `.exe`. Если пользователь добавил `chrome.exe` в VPN-зону, все запуски этого `chrome.exe` контролируются независимо от источника запуска.

Один `.exe` может быть только в одной зоне. Если пользователь пытается добавить его во вторую, действие запрещается. Сообщение: "Это приложение уже добавлено в VPN-зону / прямую зону. Удалите его из текущей зоны, чтобы добавить в другую."

При старте агент проверяет уже запущенные процессы. Если среди них есть управляемые приложения, к ним сразу применяются правила зоны.

Дочерние процессы контролируются временно, если они явно относятся к управляемому приложению. Примеры:

- браузерные helper-процессы;
- WebView/CEF;
- игровые лаунчеры;
- `steamwebhelper.exe`;
- сетевые helper-процессы, запущенные основным приложением.

Updater и crash reporter не добавляются навсегда автоматически. Все временные связи попадают в диагностику, чтобы пользователь и поддержка видели, почему процесс оказался под правилом.

Автозапуска управляемых приложений в MVP нет. Автоматически стартует только агент/контроллер.

## 5. Поведение при нарушениях

Если приложение из зоны запускается при нарушенных условиях, оно сразу завершается, а пользователь видит окно с причиной:

- какое приложение заблокировано;
- какая зона сработала;
- причина: VPN выключен, страна не совпадает, прямой маршрут не подтвержден, сетевой компонент недоступен;
- что сделать дальше: включить VPN, сменить активный профиль, выключить VPN для прямой зоны, открыть настройки или запустить мастер.

Если нарушение происходит во время работы приложения, действует настройка зоны:

- закрыть сразу;
- закрыть через 10 секунд;
- закрыть через 20 секунд;
- закрыть через 30 секунд;
- только предупреждать, если пользователь явно подтвердил риск.

Если условие восстановилось до конца таймера, закрытие отменяется. Например, если для прямой зоны пользователь выключил VPN и прямой маршрут снова совпал с baseline, приложение продолжает работать. Для VPN-зоны выключение VPN является нарушением: новые запуски блокируются, а уже запущенные приложения получают реакцию по настройке зоны.

Если WFP/native-компонент или служба работают некорректно, общий интернет пользователя не блокируется. Вместо этого зоны получают статус "контроль недоступен / требуется внимание". VPN-зона без полного контроля не запускает управляемые приложения. Прямая зона показывает, что обход не гарантирован, и действует по политике пользователя.

## 6. Интерфейс

Визуальное направление: дружелюбный системный интерфейс. Приложение должно объяснять, что произошло и что делать дальше, без лишнего технического шума.

Основные части UI:

- **Первый запуск**: мастер выбора сценария и калибровки.
- **Главный экран**: статусы активных зон, причина предупреждений, быстрый переход к настройкам.
- **Зоны**: списки только управляемых приложений. Приложения без правил не показываются.
- **Профили**: VPN-профили и прямые профили, активный профиль, переименование, удаление.
- **Журнал событий**: короткая история последних событий с временем, зоной, приложением и причиной.
- **Диагностика**: состояние службы, WFP-компонента, geo-IP, маршрутов, DNS, распознавания VPN.
- **Трей**: статус зон, открыть окно, открыть журнал, выйти с предупреждением.
- **Мини-индикатор**: компактное окно поверх всех окон. Его можно спрятать. Он показывает цветовые статусы зон и короткие пояснения, без списка приложений.

Цветовые статусы:

- зеленый: работает штатно;
- желтый: требует внимания или деградация;
- красный: условия нарушены / зона блокирует приложения;
- серый: зона отключена.

При выходе из агента пользователь видит предупреждение: "После выхода правила зон перестанут применяться."

## 7. Данные, логи и отчеты

Настройки хранятся в SQLite в системной папке вроде `%ProgramData%`. В базе лежат:

- режим работы;
- включенные и отключенные зоны;
- приложения зон;
- VPN-профили;
- прямые профили;
- активные профили;
- реакции на нарушения;
- версия схемы для миграций.

Критичные настройки защищаются Windows ACL и проверкой целостности через хэш/подпись правил. Шифрование в MVP не используется: правила зон не являются секретом, а ключ все равно находился бы на устройстве.

Логи отдельные от базы: структурированные текстовые файлы для диагностики. Приватные данные маскируются. В UI есть журнал событий, чтобы пользователь видел причины событий: "VPN сменил страну", "geo-IP недоступен", "прямой маршрут восстановлен", "Chrome заблокирован".

Диагностический отчет формируется только по инициативе пользователя:

1. Пользователь нажимает "Сформировать отчет".
2. Пользователь видит весь отчет перед отправкой.
3. Пользователь может указать название VPN/клиента и комментарий.
4. Пользователь вручную отправляет отчет.
5. После отправки приложение показывает: "Спасибо. Ваша информация помогает развитию программы."

Не собираются:

- VPN-конфиги;
- приватные ключи;
- пароли;
- токены;
- история сайтов;
- содержимое DNS-запросов;
- полный список процессов.

Отправку отчета лучше делать через серверный endpoint владельца продукта. Endpoint уже пересылает данные в Telegram или почтовый парсер. Токены Telegram не зашиваются в клиентское приложение.

## 8. Сеть, geo-IP и обновления

Проверка VPN/локации гибридная:

- внешний IP и страна через собственный endpoint как основной источник;
- публичные geo-IP сервисы как резерв;
- локальные признаки: адаптеры, маршруты, DNS, процессы VPN-клиентов, WFP-состояние;
- уровень уверенности распознавания протокола/клиента.

Внешний geo-IP не запрашивается слишком часто. Проверка выполняется при старте, сетевых изменениях, подозрении на смену VPN и по разумному cooldown. Если geo-IP недоступен, зона не считается молча валидной: статус переходит в "требует внимания".

Прямая зона проверяется гибридно:

- основной контроль через WFP/маршрут/интерфейс процесса;
- диагностический probe для подтверждения внешнего IP/маршрута;
- если обход обеспечен сторонним VPN-клиентом, UI показывает "обход подтвержден настройками VPN-клиента", но приложение не меняет настройки стороннего клиента.

Обновления:

- приложение проверяет наличие новой версии;
- показывает уведомление;
- открывает сайт или Telegram-канал;
- автоустановки в MVP нет.

Для массового распространения через `.exe`-установщик нужно учитывать подпись установщика и компонентов как важный будущий пункт.

## 9. Установка, деградация и безопасность

Распространение: один `.exe`-установщик.

Установщик:

- запрашивает права администратора;
- ставит PyQt6-приложение;
- ставит Windows service;
- устанавливает или регистрирует WFP/native-компонент;
- добавляет ярлык;
- добавляет ярлыки в меню "Пуск";
- включает автозапуск агента;
- добавляет деинсталлятор;
- регистрирует удаление через стандартные "Приложения и возможности" Windows.

Ярлыки в меню "Пуск":

- открыть "Песочница VPN";
- открыть диагностику/журнал, если это будет вынесено отдельным ярлыком;
- удалить "Песочница VPN".

Деинсталлятор должен корректно снимать службу, WFP/native-компонент, автозапуск и правила.

Главный принцип деградации: не ломать общий интернет пользователя. Если системный компонент не работает, приложение не блокирует весь трафик. Оно показывает, что полный контроль недоступен. VPN-зона без контроля не запускает управляемые приложения. Прямая зона показывает, что обход не гарантирован, и действует по политике пользователя.

Антиобход, пароли и запрет выключения в MVP не используются. Пользователь является владельцем своего компьютера.

## 10. MVP и границы первой версии

В MVP входит:

- Windows 10/11;
- PyQt6 UI;
- Windows service;
- WFP/native сетевой компонент;
- сценарии: VPN-контроль, прямой обход VPN, две зоны;
- мастер калибровки;
- несколько VPN-профилей, один активный;
- несколько прямых профилей, один активный;
- добавление приложений из найденного списка и вручную через `.exe`;
- контроль запусков из любых мест Windows;
- контроль уже запущенных процессов при старте агента;
- временная привязка дочерних процессов;
- журнал событий;
- трей;
- мини-индикатор;
- диагностический отчет по инициативе пользователя;
- проверка обновлений;
- `.exe`-установщик с меню "Пуск" и деинсталлятором.

Не входит в MVP:

- Linux;
- macOS backend;
- автообновление;
- несколько рабочих наборов зон;
- автозапуск управляемых приложений;
- правила по отдельным ярлыкам или профилям браузера;
- автоматическое изменение конфигов сторонних VPN-клиентов;
- запрет пользователю выключать агент;
- шифрование настроек.

## 11. Тестирование и проверка

Минимальная стратегия тестирования:

- unit-тесты для модели зон, профилей, реакций на нарушения и миграций SQLite;
- интеграционные тесты UI-служба через IPC;
- тесты установки, удаления и повторной установки;
- тесты деградации: служба остановлена, WFP-компонент недоступен, geo-IP endpoint недоступен;
- тесты VPN-зоны: VPN выключен, страна совпадает, страна не совпадает, geo-IP недоступен;
- тесты прямой зоны: VPN выключен, VPN включен, прямой маршрут подтвержден, прямой маршрут не подтвержден;
- тесты уже запущенных процессов при старте агента;
- тесты дочерних процессов для браузеров, WebView/CEF и игровых лаунчеров;
- ручные тесты на популярных VPN-клиентах и протоколах.

Критическое правило проверки: сбой системного компонента не должен ломать общий интернет пользователя. Он должен переводить зоны в честный статус деградации.

## 12. Открытые технические решения для этапа планирования

Перед реализацией нужно отдельно выбрать:

- язык native-компонента: C++ или Rust;
- конкретную IPC-модель;
- формат подписания/хэширования критичных правил;
- формат SQLite-схемы и миграций;
- формат структурированных логов;
- установщик: Inno Setup, WiX, NSIS или другой вариант;
- минимальный набор VPN-клиентов для ручного тестирования первой версии;
- контракт собственного geo-IP/update/report endpoint.
