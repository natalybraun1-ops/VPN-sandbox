# Product Logic And Decisions

Документ фиксирует основную логику "Песочницы VPN" и договоренности, уже
принятые в обсуждении. Он нужен как стартовый контекст для следующих веток
диалога.

## Продуктовая цель

"Песочница VPN" - бесплатное Windows-приложение, которое помогает пользователю
контролировать, какие выбранные программы должны работать только при валидном
VPN, а какие должны идти напрямую в обход VPN.

Продукт не планируется коммерческим сам по себе. Он должен быть бесплатной
полезной утилитой и помогать продвижению платного софта владельца проекта.

## Платформы

- Основная платформа MVP: Windows 10/11.
- Linux не планируется.
- macOS может быть позже, но не входит в MVP.
- Архитектура должна по возможности сохранять переносимую бизнес-логику, чтобы
  будущий macOS backend не требовал переписывать весь продукт.

## Главная модель зон

В приложении есть один набор настроек и до двух зон.

### VPN-зона

Приложения в VPN-зоне могут запускаться и работать только при выполнении правил
VPN-зоны.

Строгие условия:

- VPN обнаружен;
- страна совпадает с активным VPN-профилем.

Город или регион показывается только как информационный маркер. Город не
влияет на блокировку, предупреждение или разрешение запуска.

### Прямая зона

Приложения в прямой зоне должны работать напрямую, без VPN. Если VPN включен,
прямая зона должна проверять, что для управляемого приложения есть подтвержденный
прямой маршрут или обход через настройки VPN-клиента.

Если пользователь выключает VPN, работа прямой зоны должна восстанавливаться
автоматически, а VPN-зона при этом должна перейти в блокировку или предупреждение
по своим правилам.

### Режимы первого запуска

Пользователь выбирает один из трех сценариев:

- только VPN-контроль;
- только прямая зона;
- две зоны.

Отключенная зона не применяет правила, не показывает предупреждения и не мешает
пользователю. Ее настройки сохраняются и могут быть включены позже.

## Управляемые приложения

- Пользователь добавляет приложения из списка найденных программ или вручную
  через выбор `.exe`.
- Приложения запускаются пользователем обычным способом: с рабочего стола,
  панели задач, меню "Пуск", ярлыков или из другого приложения.
- Программа не должна становиться обязательным лаунчером.
- Неуправляемые приложения полностью игнорируются: они не показываются как
  "не контролируется" и не создают предупреждений.
- Один `.exe` может находиться только в одной зоне. Если пользователь пытается
  добавить его во вторую зону, действие запрещается с понятным сообщением.
- Для MVP правило применяется ко всему `.exe`, а не к отдельному ярлыку или
  профилю браузера.

## Дочерние процессы

Если приложение создает явно связанные дочерние процессы, их нужно временно
контролировать вместе с основным приложением. Это важно для браузеров, WebView,
CEF, игровых лаунчеров и помощников вроде `steamwebhelper.exe`.

Цель: не допустить ситуации, когда основное приложение контролируется, а его
сетевые helper-процессы обходят правила зоны.

## Реакция на нарушение

Реакция на нарушение настраивается отдельно для каждой зоны.

Варианты:

- закрыть сразу;
- закрыть через 10 секунд;
- закрыть через 20 секунд;
- закрыть через 30 секунд;
- только предупредить.

Режим "только предупредить" должен требовать явной галочки/подтверждения, что
пользователь понимает риск. Без такого подтверждения `WARN_ONLY` не должен
молча превращаться в разрешение.

## Деградация и безопасность

Главный принцип деградации: приложение не должно ломать общий интернет
пользователя.

Если системный сетевой компонент, WFP-контроль, geo-IP или другой важный
механизм недоступен:

- общий интернет не блокируется;
- управляемые зоны получают честный статус "требует внимания" или "контроль
  недоступен";
- VPN-зона без контроля не должна молча считать условия валидными;
- прямая зона должна предупреждать, что обход не гарантирован, и действовать
  по пользовательской политике.

Администраторские права для установки и системного контроля считаются
нормальными для этого продукта.

## Калибровка и профили

Настройка должна идти через мастер калибровки.

### VPN-профиль

Для VPN-зоны пользователь включает нужный VPN, а приложение сканирует текущее
состояние:

- внешний IP;
- страна;
- город или регион как информационный маркер;
- вероятный VPN-клиент;
- вероятный протокол;
- сетевой адаптер;
- DNS и маршруты;
- уровень уверенности распознавания.

Пользователь видит результат вида: VPN/клиент, протокол, страна, город. Затем
подтверждает сохранение.

Можно хранить несколько VPN-профилей, но активным для зоны является только один.
Остальные являются запасными и не активируются автоматически. Пользователь может
переименовать или удалить профиль.

### Прямой профиль

Для прямой зоны калибровка делается без VPN. Это надежнее, чем пытаться угадать
прямой маршрут при включенном VPN.

Можно хранить несколько прямых профилей для разных сетей, например дом, офис,
мобильный интернет. Активный профиль выбирается пользователем вручную.

## VPN-протоколы и распознавание

Нужно широкое распознавание популярных и обходных VPN/туннельных решений:

- WireGuard;
- OpenVPN;
- IKEv2;
- L2TP;
- PPTP;
- SSTP;
- Outline;
- Amnezia;
- Shadowsocks;
- Clash;
- V2Ray/Xray;
- похожие туннельные клиенты.

Редкие случаи можно определять предположительно. Если протокол не определяется,
пользователь может вручную отправить отчет разработчику.

## Отчеты и приватность

Отчет формируется и отправляется только по инициативе пользователя.

Процесс:

1. Пользователь нажимает "Сформировать отчет".
2. Пользователь видит весь отчет перед отправкой.
3. Пользователь может указать название VPN/клиента и комментарий.
4. Пользователь вручную подтверждает отправку.
5. После отправки приложение показывает благодарность.

Не собирать:

- VPN-конфиги;
- приватные ключи;
- пароли;
- токены;
- историю сайтов;
- содержимое DNS-запросов;
- полный список процессов.

Отправку лучше делать через серверный endpoint владельца продукта, который уже
пересылает данные в Telegram или почтовый парсер. Telegram-токены не должны быть
зашиты в клиентское приложение.

## UI-договоренности

- UI строится на PyQt6.
- Мастер первого запуска уже реализован: пользователь выбирает VPN-зону,
  прямую зону или две зоны, а выбор сохраняется в SQLite.
- Основное окно уже реализовано как PyQt6 shell с вкладками "Обзор", "Зоны",
  "Профили", "Приложения", "Журнал", "Диагностика".
- Настройки зон уже редактируются через UI на основе `ZoneSettings`.
- UI профилей уже умеет вручную добавлять VPN/direct профили, выбирать активный
  профиль и удалять профиль. Этап 2 должен подключить к этому существующему UI
  реальные мастера калибровки, а не переписывать профильные таблицы с нуля.
- Список управляемых приложений уже умеет ручное добавление `.exe` и удаление
  выбранного приложения.
- Нужен журнал событий, чтобы пользователь понимал, что происходит: например
  VPN сменил страну, geo-IP недоступен, прямой маршрут восстановлен, приложение
  заблокировано.
- Tray уже реализован: открыть окно, открыть журнал, показать mini-indicator,
  выйти из приложения. Exit должен закрывать mini-indicator и завершать Qt app.
- Mini-indicator уже реализован как компактное окно поверх остальных окон.
- В мини-индикаторе достаточно цветов и пояснения: "работает штатно" или
  "требует внимания". Список приложений там не обязателен.
- Не показывать "Telegram не контролируется" и подобные статусы для приложений,
  которые пользователь не добавлял в зоны.

## Установка

Распространение MVP: один `.exe`-установщик.

Установщик должен:

- запросить права администратора;
- поставить PyQt6-приложение;
- поставить Windows service;
- поставить или зарегистрировать native/WFP-компонент;
- добавить ярлык;
- добавить пункты в меню "Пуск";
- включить автозапуск агента по умолчанию;
- добавить деинсталлятор;
- зарегистрировать удаление через стандартные настройки Windows.

Деинсталлятор должен корректно снимать службу, WFP/native-компонент, автозапуск,
ярлыки и правила.

## Текущая архитектура кода

Код на ветке `codex/pyqt-shell` состоит из двух готовых слоев.

### Foundation

Реализован Python-фундамент:

- модели зон, профилей, настроек зон и управляемых приложений;
- policy evaluation для VPN-зоны и прямой зоны;
- SQLite-хранилище VPN-профилей, direct-профилей, настроек зон и приложений;
- запрет одного `.exe` в двух зонах;
- JSONL-журнал событий с рекурсивной маскировкой IP;
- IPC message contracts;
- локальный service simulator для раннего UI и тестов;
- CLI smoke-команда `python -m vpn_sandbox doctor`.

### PyQt6 Application Shell

Реализована первая пользовательская оболочка:

- `src/vpn_sandbox/app/bootstrap.py` собирает SQLite repository, journal и
  application context;
- `src/vpn_sandbox/app/controller.py` является тонким application/service слоем
  между PyQt widgets, repository, journal и `ServiceSimulator`;
- `src/vpn_sandbox/app/paths.py` поддерживает `VPN_SANDBOX_DATA_DIR` для
  локальных запусков и тестов;
- `src/vpn_sandbox/ui/app.py` запускает PyQt6 приложение;
- `src/vpn_sandbox/ui/first_run.py` отвечает за первый запуск;
- `src/vpn_sandbox/ui/main_window.py` содержит основной shell и вкладки;
- `src/vpn_sandbox/ui/tray.py` и `src/vpn_sandbox/ui/mini_indicator.py`
  реализуют tray и компактный индикатор;
- `src/vpn_sandbox/ui/view_models.py`, `text.py`, `widgets.py` держат
  форматирование и небольшие UI-помощники отдельно от доменной логики.

Важно: PyQt widgets не должны напрямую работать с SQLite, файловым журналом,
сетевым сканированием или процессами. Новая логика Stage 2 должна входить через
application/controller слой и покрываться тестами без обязательного запуска GUI.

## Проверка работоспособности

Полный suite:

```powershell
$env:QT_QPA_PLATFORM = "offscreen"
python -m pytest -p no:cacheprovider --basetemp .pytest-tmp -q
```

Smoke-команда:

```powershell
$env:PYTHONPATH = "src"
python -m vpn_sandbox doctor
```

Ожидается JSON с `"storage_schema": 2`.

Запуск UI из checkout:

```powershell
$env:PYTHONPATH = "src"
$env:VPN_SANDBOX_DATA_DIR = "$PWD\.local-data"
python -m vpn_sandbox ui
```

## Что еще не реализовано

Пока нет:

- реальной калибровки VPN/direct профилей;
- определения внешнего IP, страны, города, сетевых адаптеров, маршрутов, DNS и
  процессов VPN-клиентов;
- Windows service;
- WFP/native-компонента;
- real process interception/enforcement;
- geo-IP/update/report endpoint;
- installer/uninstaller.

Этап 2 должен начать с Calibration And Detection и не переходить к системному
контролю процессов или WFP.
