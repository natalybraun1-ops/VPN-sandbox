# Global Roadmap

Этот roadmap задает крупные этапы разработки "Песочницы VPN". Он не должен
превращаться в детальный task-list. Следующие ветки диалога выбирают один
глобальный этап и уже для него создают подробный план реализации.

## Этап 0. Python Foundation

Статус: готово в ветке `codex/foundation`.

Смысл этапа: зафиксировать доменную модель, правила зон, хранилище, журнал,
IPC-контракты, локальный симулятор сервиса и smoke-команду.

Проверка:

```powershell
python -m pytest
$env:PYTHONPATH = "src"
python -m vpn_sandbox doctor
```

## Этап 1. PyQt6 Application Shell

Статус: готово в ветке `codex/pyqt-shell`.

Цель: сделать первое пользовательское приложение без системного контроля.

Глобальный результат:

- главное окно;
- мастер первого запуска;
- выбор сценария: VPN-зона, прямая зона, две зоны;
- экраны настроек зон;
- экраны профилей;
- список управляемых приложений;
- журнал событий;
- tray;
- мини-индикатор поверх окон или скрываемый индикатор.

На этом этапе UI может работать с локальным `ServiceSimulator`, без Windows
service и WFP.

## Этап 2. Calibration And Detection

Цель: научить приложение создавать реальные VPN-профили и прямые профили.

Глобальный результат:

- мастер калибровки VPN при включенном VPN;
- мастер калибровки прямого профиля без VPN;
- определение внешнего IP, страны, города;
- локальные признаки VPN: адаптеры, маршруты, DNS, процессы клиентов;
- уровень уверенности распознавания;
- хранение нескольких профилей;
- ручное переименование, выбор активного профиля, удаление профиля;
- понятная деградация при недоступном geo-IP.

На этом этапе нельзя автоматически менять настройки сторонних VPN-клиентов.

## Этап 3. Local IPC And Background Agent

Цель: разделить UI и фоновую логику.

Глобальный результат:

- фоновый агент в пользовательской сессии или прототип службы;
- защищенный локальный IPC;
- UI отправляет настройки и получает статусы;
- агент проверяет уже запущенные управляемые процессы при старте;
- агент пишет события в журнал;
- агент запускается автоматически.

Этот этап еще может не иметь полноценного WFP-контроля, но должен подготовить
контракт для будущего Windows service.

## Этап 4. Process Control

Цель: применить правила зон к реальным процессам Windows.

Глобальный результат:

- обнаружение запуска управляемого `.exe`;
- проверка уже запущенных процессов при старте агента;
- контроль связанных дочерних процессов;
- предупреждения и таймауты закрытия;
- режим "только предупредить" только после подтверждения пользователя;
- понятные события в журнале;
- неуправляемые приложения полностью игнорируются.

На этом этапе важно не строить обязательный лаунчер. Пользователь продолжает
запускать приложения обычным способом.

## Этап 5. Native Network Control

Цель: добавить настоящий системный сетевой контроль.

Глобальный результат:

- Windows service;
- native/WFP-компонент;
- привязка сетевых потоков к процессам;
- контроль VPN-зоны по VPN/стране;
- контроль прямой зоны по прямому маршруту или подтвержденному обходу;
- защита от утечек управляемых приложений;
- fail-open для общего интернета;
- честный статус деградации, если контроль недоступен.

Этот этап требует отдельного технического выбора: C++ или Rust для native-части,
модель WFP-правил, установка драйверов/служб, подпись компонентов.

## Этап 6. Diagnostics, Reports, Updates

Цель: сделать поддержку и развитие продукта безопасными для пользователя.

Глобальный результат:

- UI-журнал событий;
- диагностический отчет с preview перед отправкой;
- поле для названия VPN/клиента и комментария пользователя;
- отправка отчета только по ручному подтверждению;
- серверный endpoint для пересылки в Telegram или почтовый парсер;
- отсутствие Telegram-токенов в клиенте;
- проверка обновлений без автоустановки в MVP.

## Этап 7. Installer And Uninstaller

Цель: подготовить массовое распространение.

Глобальный результат:

- `.exe`-установщик;
- запрос admin rights;
- установка UI, агента/службы и native/WFP-компонента;
- автозапуск агента по умолчанию;
- ярлыки и меню "Пуск";
- деинсталлятор;
- удаление через стандартные настройки Windows;
- корректное снятие службы, WFP/native-компонента, автозапуска и правил.

Отдельно нужно решить вопрос подписи установщика и компонентов.

## Этап 8. QA And Hardening

Цель: проверить продукт на реальных пользовательских сценариях.

Глобальный результат:

- тесты деградации: служба остановлена, WFP недоступен, geo-IP недоступен;
- ручные тесты популярных VPN-клиентов и протоколов;
- тесты VPN-зоны: VPN выключен, страна совпадает, страна не совпадает;
- тесты прямой зоны: VPN выключен, VPN включен, прямой маршрут есть/нет;
- тесты дочерних процессов для браузеров, WebView/CEF, игровых лаунчеров;
- проверка установки, удаления и повторной установки;
- проверка, что общий интернет не ломается при сбоях.

## Этап 9. Public Beta And Distribution

Цель: подготовить бесплатную публичную раздачу.

Глобальный результат:

- страница скачивания или сайт;
- публикация в Telegram-канале;
- инструкция первого запуска;
- известные ограничения MVP;
- механизм сбора добровольных отчетов;
- список поддерживаемых VPN-клиентов и протоколов;
- процесс выпуска обновлений.

## Этап 10. Future macOS Backend

Статус: не входит в MVP.

Цель: сохранить возможность будущей macOS-версии без переписывания всей
бизнес-логики.

Глобальный результат в будущем:

- отдельный macOS backend;
- переносимый UI/модель зон там, где это рационально;
- Network Extension, Packet Filter или другой системный механизм macOS;
- отдельная стратегия установки и прав.

Linux не планируется.

## Как работать дальше

Для следующей ветки диалога:

1. Выбрать один глобальный этап.
2. Проверить, что он не требует незакрытого предыдущего этапа.
3. Создать отдельный детальный implementation plan.
4. Реализовывать через отдельную feature branch/worktree.
5. После реализации обновить этот roadmap, если глобальная карта изменилась.
